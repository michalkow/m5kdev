# Managed consumer updates in 0.31.0

## Summary

`create-m5kdev@0.31.0` introduces `.m5kdev.json`, `init`, `doctor`, and
`update`. Newly scaffolded apps record their template context, enabled
features, managed catalog, file policies and hashes, and applied migration IDs.
The CLI can then reconcile later templates without becoming an application
dependency.

Applications created before 0.31.0 are **not** historically migration-supported.
They may enroll at the 0.31.0 baseline only after being brought to a compatible
framework, catalog, and starter structure.

## Who needs this migration

- Apps created with 0.31.0 or later need no bootstrap action; scaffolding writes
  `.m5kdev.json` automatically.
- Existing apps that want managed updates must first align manually with the
  0.31.0 framework and starter, then run `init`.
- Apps that do not want managed updates may continue upgrading manually.

The production migration registry is intentionally empty at this baseline.
The guide ID identifies the bootstrap documentation, not a backfilled transform.

## Prerequisites and upgrade order

1. Use Node.js and pnpm versions supported by the repository.
2. Commit or stash application changes before a write-mode update.
3. Bring a pre-0.31 app to the tested framework/catalog state manually.
4. Run `doctor`, enroll with `init`, preview with `update --dry-run`, then apply.

Always invoke an exact CLI release:

```sh
pnpm dlx create-m5kdev@0.31.0 doctor
```

Do not add `create-m5kdev` to the generated application's dependencies.

## What changed

- `.m5kdev.json` stores schema version 1, template name/version, non-secret
  rendering context, features, the managed catalog snapshot, baseline hashes
  and file policies, and applied migration IDs.
- `m5kdev init` establishes the running CLI version as the baseline. It does not
  modify application files or replay older migrations.
- `m5kdev doctor` provides stable static diagnostics; `--full` also runs
  available `check-types`, `lint`, and `build` scripts.
- `m5kdev update --dry-run` performs the complete plan without writing.
- Write-mode updates require clean Git and apply only conflict-free plans.

## Dependency and catalog changes

The generated consumer catalog is now derived from every starter variant and
the monorepo's tested catalog. Framework versions come from their package
manifests. The previously drifted entries are corrected:

| Entry | Previous template | 0.31.0 generated value |
| --- | --- | --- |
| `@m5kdev/backend`, `commons`, `config`, `email`, `frontend`, `web-ui` | `0.26.2` | `0.30.0` |
| `@libsql/client` | `0.17.0` | `0.17.4` |
| `@types/react` | `19.2.7` | `19.2.17` |
| `drizzle-kit` | `0.31.4` | `0.31.10` |
| `drizzle-orm` | `0.44.3` | `0.45.2` |
| `drizzle-zod` | `0.8.2` | `0.8.3` |

The updater changes an unchanged managed value, preserves app-owned catalog
entries, adds new managed entries, and removes obsolete unchanged entries. A
locally customized managed value is a conflict and is never overwritten.

## Starter and template file changes

| Path | Existing app action | Policy |
| --- | --- | --- |
| `.m5kdev.json` | Add with `init` after compatibility checks | CLI-owned state |
| `pnpm-workspace.yaml` | Align the managed catalog before enrollment | Structural merge |
| `package.json` and app manifests | Validate all `catalog:` references | Three-way merge |
| `biome.json` | Accept the generated template's LF line endings | Three-way merge |
| Text starter/root-template files | Keep custom changes; review future plans | Three-way merge |
| Binary template files | Replace only when locally unchanged | Binary-safe |
| Real `.env` files | None; never reconciled or re-rendered | Ignored |
| `**/drizzle/*.sql` | None | Ignored |
| `**/drizzle/meta/**` | None | Ignored |

Feature-disabled template paths are excluded from baselines and updates.
Declared additions, deletions, and renames are planned explicitly. Managed
symlinks and paths escaping the repository are rejected.

## Migration steps

1. **Manual:** update the pre-0.31 app to compatible `@m5kdev/*` packages and
   exact catalog values shown above. Review the current starter for required
   files and composition changes.
2. **Manual:** install and verify the compatible state:

   ```sh
   pnpm install
   pnpm check-types
   pnpm lint
   pnpm build
   ```

3. **Automatic:** inspect the app and establish the 0.31.0 baseline:

   ```sh
   pnpm dlx create-m5kdev@0.31.0 init
   ```

   Use `--yes` only for deliberate non-interactive enrollment. Existing state
   is refused unless `--force` is supplied. `--force` re-baselines; it does not
   repair or modify app files.
4. **Automatic:** run diagnostics and preview the full plan:

   ```sh
   pnpm dlx create-m5kdev@0.31.0 doctor --full
   pnpm dlx create-m5kdev@0.31.0 update --dry-run
   ```

5. **Automatic:** for a later release, use that exact version to apply a clean,
   conflict-free plan:

   ```sh
   pnpm dlx create-m5kdev@<target-version> update
   ```

   Dependency-input changes trigger `pnpm install`. With `--skip-install`, run
   the printed install command yourself.

## Database migration

No database schema or data migration is introduced by the managed updater.
Drizzle SQL history and metadata snapshots are deliberately outside template
reconciliation. Future schema migrations must still be generated using the
application's Drizzle generation command; do not create those files by hand.

## Environment and deployment changes

No new runtime environment variable or service is required. `.m5kdev.json`
contains no auth secret or other secret rendering value and should be committed.
The updater may fetch the baseline `create-m5kdev@<version>` artifact from npm
only when a locally modified text file requires its original merge base.

## Verification

```sh
pnpm dlx create-m5kdev@0.31.0 doctor --full
git diff --check
git status --short
```

`doctor` succeeds when there are no error diagnostics. Warnings alone return a
successful status. Legitimate edits to merge-managed files are informational.
For automation, add `--json` to `init`, `doctor`, or `update` and consume the
stable diagnostic/change fields.

## Rollback

Before enrollment, rollback consists of reverting catalog and compatibility
edits. After enrollment, `.m5kdev.json` can be reverted with the rest of the
commit. An update builds its entire source plan before writing; conflicts cause
zero repository writes. If `pnpm install` fails after a successful application,
the source and state remain applied and idempotent: fix registry/network issues
and rerun `pnpm install`.

Database rollback is not applicable to this release.

## Troubleshooting

- **`init` reports errors:** finish the 0.31 compatibility work. `init` permits
  warnings but does not establish a known-bad baseline.
- **Historical artifact unavailable or corrupt:** retry with registry access or
  resolve the affected file manually; the repository is not written.
- **Update requires clean Git:** commit or stash changes, then rerun. Doctor and
  dry-run remain available on dirty trees.
- **Three-way conflict:** preserve the app's intent, reconcile it with the target
  template, commit the resolution, and rerun the update.
- **CLI is older than state:** invoke the state version or a newer exact release.

## Complete upgrade checklist

- [ ] Align all managed framework and third-party catalog entries.
- [ ] Compare required and feature-specific starter/root-template paths.
- [ ] Run install, typecheck, lint, and build.
- [ ] Commit the compatible pre-enrollment state.
- [ ] Run `pnpm dlx create-m5kdev@0.31.0 init` and commit `.m5kdev.json`.
- [ ] Run `doctor --full` successfully.
- [ ] Run the target version's `update --dry-run` and review every change.
- [ ] Apply only from a clean worktree and resolve no conflicts by overwriting.
- [ ] Run `pnpm install` if it was skipped or failed.
- [ ] Re-run doctor and application verification after the update.
